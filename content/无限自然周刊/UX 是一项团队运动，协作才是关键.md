UX 是一项团队运动：协作才是关键


我们经常关注线框图、用户研究，或者像 Figma 这样的设计工具，但==如果我们无法良好协作，这些都无法真正推动项目前进==。优秀的 UX 不是孤立完成的，它需要与工程师对话、与产品、销售和其他利益相关者保持一致，还需要倾听、适应和共同创造的能力。==这正是设计变成团队运动的时刻==，而你==记录与推动这些协作成果的能力，会成倍放大 UX 的影响力==。

  

当人们谈论 UX 时，通常关注那些可见和可交互的内容，比如线框图和原型、聪明的交互设计、以及 Figma、Miro、Maze 等设计工具。有些输出甚至被过度美化，比如设计系统、研究报告以及像素级别精致的 UI 设计。

  

但在我 20 多年的 UX 工作中反复验证的一个真相是：==如果没有协作，所有这些都不会真正产生价值==。

  

优秀的 UX 并不是孤独完成的。它诞生于与工程师、产品经理、客户团队、处理工单的客服团队之间的对话。令人惊艳的 UX 灵感，往往出现在混乱的 Miro 工作坊、跨职能的研讨会、或者 Slack/Teams 的日常对话中 —— 那些人们彼此对齐、灵活调整并共同创造的地方。

  

我职业生涯中最有影响力的一些时刻，==往往不是“传统意义上”的设计时刻==，而是与经验各异的团队成员一起讨论问题、头脑风暴，诞生出我一个人无法想出的好点子。正如我常说的，==十个人在一个房间里，能提出的点子是一个人的十倍==，其中很多想法就是最终最有价值的结果。

  

有些时候，团队在工作坊中帮助我重新框定了问题，从模糊甚至相互矛盾的反馈中理清思路，找到明确方向；又或者，我坐在销售代表旁边，听他讲起反复出现的用户投诉。这些，==才是设计变成团队运动的时刻==，==而你捕捉这些成果的能力，就是 UX 
  
  

# 为什么这篇文章此刻尤为重要？

  

  

现在我们对“协作”的关注之所以迫切，是因为疫情以来的工作方式发生了巨大变化。==根据美国劳工部的一项研究，团队变得更加跨职能、远程和复杂==。因为物理距离和缺乏面对面交流，==更容易形成信息孤岛==；但与此同时，==对齐的必要性却比以往任何时候都更重要==。

  

==协作不再是“锦上添花”，而是 UX 核心能力之一==，因为我们的工作涉及组织中多个部门、多个触点。

  

  

  

  

# 什么才是真正的 UX 协作？

  

  

首先，澄清一个误解：==协作不等于配合==。

  

- 配合：你做你的，我做我的，之后再同步。
- 协作：我们一起解决问题，并共同承担结果。

  

  

在 Deakin 大学出版的《Communication Concepts》中，协作被定义为：与他人共同完成产出或达成目标。协作的结果往往是一个可见的产品，或者一个明确的成果，比如解决问题或做出决策。

  

  

案例：B2B 欺诈预警平台设计

  

  

最近我参与了一个为金融科技公司设计欺诈预警平台的六个月项目。我们完全无法接触到终端用户，因为产品尚未上市，且用户是非常专业的 B2B 财务人员，难以找到。

  

团队成员分布在马来西亚、墨尔本，而我在悉尼。这看似是死局，但我们转而“向内”协作：找来了主题专家、合规顾问、客服团队等，他们对欺诈模式与用户痛点有深刻理解。

  

我们每两周开展一次 Miro 工作坊，进行反复反馈与草图迭代。我甚至邀请他们自己画出设计方案。

  

通过这样的协作方式，我们最终设计出了两个不同版本的调查仪表盘框架。我没有直接选一个最好的去推动，而是邀请 PM、工程师、SME、客服等一起投票。每个人都有发言权，最终胜出的设计正是大家一起创造并验证的成果。==这是一个全员共创、解决真实用户问题的典范案例==。

  

这也是我职业中最有满足感的项目之一。

  

  

  

  

失败的协作，带来的教训

  

  

我还和一位老同事聊天，她现在是某产品的负责人。她提到一个失败的例子：设计团队完成重大改版后，才让她参与评审。结果显而易见 —— 设计忽略了几个关键的产品限制和商业目标，不得不推翻重来。这再次证明，==最好的设计永远不是闭门造车==。

  

  

  

  

真正协作，是跨越角色边界的

  

  

协作不仅仅是设计师和 PM 之间的事情。QA 测试人员、内容策略师、销售代表、市场人员、客服等，都能带来独特洞察。

  

==最好的成果往往来自多视角融合==，而不是单一角色完成的闭环设计。

  

  

  

  

# 为什么协作经常被忽视？

  

  

协作如此关键，为什么我们却很少谈论？

  

一个原因是：==“孤胆 UX 英雄”的神话==。许多人入行是被“设计天才改变产品”的故事打动，我们的作品集也倾向于展示自己的流程与成果。而很多招聘 JD 更注重工具熟练度与交付物，而不是软技能或团队协作。

  

此外，一些企业文化推崇“快速完成任务”，这让协作显得效率低、冗余。也有一些设计师存在完美主义和领地意识 —— “这是我的设计” —— 这会扼杀协作所需的开放氛围。

  

  

  

  

# 有时候，协作就是用户研究 

  

  

理想情况下，我们总能接触用户。但现实往往是：

  

- 预算有限
- 时间紧张
- 接触路径复杂

  

  

在这些限制下，==与用户直接交流的替代方案，就是与“接触用户的人”对话==：销售、客服、技术支持、线下运维等。

  

他们就是“隐藏的用户研究员”！

  

  

另一个案例：B2C 身份验证工具的用户引导重设计

  

  

我们要重设计一个线上身份验证工具的引导流程，但因预算和时间限制，无法直接采访用户。

  

我转向销售和技术支持团队，开展多个小型研讨会，挖掘他们听到的典型问题。结果发现：==大多数用户在上传证件前就流失了==，原因可能是流程说明不清楚、不了解所需时间、不知道步骤。

  

这个发现彻底改变了我们对问题的理解，最终我们重设计了整个引导流程，加入了更清晰的说明、时间预期、操作步骤等。

  

==这类协作，其实也是一种用户研究==。虽然不能替代用户访谈，但在资源有限时，是非常有价值的补充方式。

  

  

  

  

# AI 工具来了，协作还重要吗？ 

  

  

好问题！AI 的确在创意生成、模式识别和快速原型方面发挥越来越多作用。但==AI 不会取代协作，只会改变协作的形式==。

  

AI 可以快速帮你生成设计模式、用户流程、页面布局，但它只是加速器，而非决策者。就像战略顾问 Nathan Waterhouse 所说：==AI 可以给你方向，但不能告诉你哪个方向适合你的业务与用户==。

  

你仍然需要：

  

- 客服团队来验证 AI 的方案是否真能解决问题
- 工程师评估技术可行性
- PM 确保符合业务目标

  

  

AI 只是原材料，==而协作才是将它变成“真东西”的关键==。

  

  

  

  

# 如何提升你的 UX 协作能力？ 

  

  

协作是可以练习的技能。以下是几个实用方法：

  

- 保持对他人工作的好奇心  
    多问工程师他们担心的问题，了解 PM 看重的指标，熟悉客服收到的典型问题。==你对他们的问题越关心，他们也越会关心你的设计==。
- 练习引导技巧  
    你不需要成为设计冲刺大师，只要学会主持对话、对齐目标、整合观点就很有帮助。哪怕只是简单的“工作中什么有效？什么无效？”复盘，也能带来价值。
- 早分享、常分享  
    不要等设计打磨完才展示。==草图和低保真原型更能激发协作==，大家更愿意参与，也更愿意为结果负责。
- 积极倾听  
    当有人批评你的设计，先别急着辩解。试着追问、换角度理解。==协作不是求共识，而是寻找能兼顾多种观点的方向==。
- 共同拥有结果  
    放下“这是我的设计”的执念。==最好的 UX 成果，往往不是“你的”，而是“大家的”==。

  

  

  

  

  

# 结语：UX 是一项团队运动 

  

  

==伟大的设计不会凭空诞生==，它来自开放的对话、跨职能的理解、以及共同致力于为用户解决真实问题的团队合作。

  

如果我能告诉每一个刚入行的设计师一件事，那就是：

  

==协作不是可选项，而是产出一切有意义设计的引擎==。

而对于资深设计师，它更是把好团队变成伟大团队的超级能力。

  

所以下次你准备低头“独自完成设计”时，请停下来思考：

  

- 谁还应该加入这个讨论？
- 谁能参与创造而不是仅仅审阅？

  

  

==因为最好的 UX，不是你一个人设计出来的。是大家一起共同创造的。==
